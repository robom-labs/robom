# 운영 프로토콜

## 정본과 증분 확인

- 회의 목표, 프롬프트, 제품 기준, 코드, 결정 문서, PR과 배포 기록의 정본은 GitHub다. 영구 로컬 사본은 기준으로 삼지 않는다.
- 새 대화의 첫 작업이거나 운영 설정 기준 커밋이 바뀌었을 때만 `PROTOCOL.md`, `COMPANY-MODE.md`, 해당 회의실 프롬프트를 완독한다. 같은 기준 커밋은 다시 읽지 않고 변경 여부만 확인한다.
- 요청과 직접 연결된 앱 등록 정보, 상태 문서, PR, 코드와 테스트만 확인한다. 전체 저장소 탐색, 같은 검색·파일·대형 로그의 반복 읽기는 기본값이 아니다.
- 관련 코드·설정이 바뀌지 않았다면 같은 테스트, 빌드, 의존성 설치를 반복하지 않는다. PR의 head SHA가 같으면 전체 diff도 다시 읽지 않는다.

## 01~03 총괄 권한

- 01, 02, 03은 상하 관계가 없는 동등한 총괄 부서다. 각 팀은 `robom`, `outbom`, `homebom`, `runningbom`, 관련 웹·PWA·Android, 브랜드·UX, 데이터·알림·수익화, 정책·보안, QA·CI·출시와 운영 체계를 요청 범위 안에서 다룬다.
- 사용자가 세 방에 서로 다른 일을 맡기면 서로의 허가나 완료를 기다리지 않는다. 같은 목표의 열린 PR이 있으면 중복 구현 대신 검토·보완·QA를 우선한다.
- 코드·설정 변경은 대상 저장소의 분리 브랜치와 PR로 진행하고 `main`에 직접 push하지 않는다. 01~03의 새 브랜치는 각각 `r01/`, `r02/`, `r03/` 접두사를 우선 사용하며, 동시에 작업할 때는 별도 worktree를 사용한다.

## 공유 회의록

- 두 AI가 이어 읽을 사용자 메시지와 최종 답변만 비공개 GitHub `robom-labs/ai-meeting-logs`의 `main` 브랜치 `<회의실>/`에 append-only Markdown 파일로 직접 추가한다.
- 매 답변 전 상대 AI가 마지막 확인 이후 남긴 새 기록만 읽는다. 과거 회의록은 현재 작업 ID, PR, 저장소, 기능 또는 오류와 직접 연결될 때만 찾는다.
- 기록 전 저장소 소유자와 비공개 여부를 확인한다. 회의록에는 비밀값, 개인정보, 내부 추론, 터미널 원문을 넣지 않는다.

## 실행 요청과 안전 경계

- `업데이트`, `수정`, `구현`, `적용`, `업그레이드`, `배포`, `논스톱` 같은 실행 요청은 현재 요청 범위의 조사, 구현, 검증, PR, 병합, 기존 배포 파이프라인 실행과 배포 후 검증까지 연속 수행하라는 승인으로 해석한다.
- 필수 CI, 브랜치 보호, 외부 플랫폼의 필수 검토는 우회하지 않는다. 실패하면 원인을 좁혀 수정하고 관련 검사를 다시 실행한다.
- 새 유료 결제·광고비 집행, 계정·소유권·청구 변경, 비밀값 공개·교체, 백업 없는 대량 삭제, 파괴적 마이그레이션, 새 법적 계약·스토어 약관, 신원 확인이 필요한 외부 행위는 별도 사용자 판단이 필요하다.
- 배포 후 빈 화면, 핵심 API 장애, 로그인 불가, 데이터 훼손 위험 같은 중대한 회귀가 확인되면 이전 정상 버전으로 롤백하고 원인을 보고한다.

## 검증과 보고

- 변경 중에는 직접 관련 테스트, 타입·린트, 관련 빌드, 실제 모바일·브라우저·API 확인 순으로 가장 작은 검사를 우선한다. 통합·배포 직전에 필요한 전체 검사를 한 번 실행한다.
- 비단순 요청은 독립성이 있을 때만 구현·QA·위험검토를 나누며, 같은 파일의 동시 수정과 인원 수를 위한 중복 조사는 금지한다.
- 최종 보고에는 완료 내용, 핵심 파일, PR·병합·배포 상태, 실제 검증, 롤백 여부, 남은 위험만 간결하게 적는다.
