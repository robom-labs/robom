# 디자인 결정 기록 (V6)

## 사용 도구 정직 보고

- **사용함**: Playwright(Chromium) 화면 캡처·E2E·뷰포트 스윕, node:test, ESLint(사이트), WebSearch(경쟁 패턴·WCAG 조사).
- **사용 불가(미사용)**: Figma·Canva·Claude Design·Lighthouse·axe — 이 실행 환경에 연결되어 있지 않음.
  외부 페이지 직접 열람(WebFetch)은 네트워크 정책으로 차단되어 검색 요약과 학습 지식 기반으로 조사함.
  사용하지 않은 도구를 사용했다고 보고하지 않는다.

## 비교한 디자인 방향 (3안)

### A안 — 페이지형 마법사 (화면을 단계마다 전환)
- 장점: 한 화면 정보량 최소, 단계별 집중.
- 단점: 70대 사용자가 "지금 몇 번째인지" 맥락을 잃기 쉬움. 뒤로 가기 시 입력 유지 구현이 복잡.
  화면 전환이 많아 느리게 느껴짐(젊은 사용자). `다음` 버튼 반복 = v6 계약이 금지하는 가입식 마법사.

### B안 — 아코디언 시트 (한 시트, 완료 단계는 한 줄로 접힘) ← **채택**
- 장점: 완료한 선택이 항상 화면 위에 요약으로 남아 **기억에 의존하지 않음**(S-6).
  현재 질문 하나만 크게 펼쳐져 과밀하지 않음(S-3). 뒤로(=변경)가 자연스럽고 입력이 유지됨.
  단일 스크롤 + sticky 저장으로 CTA가 사라지지 않음. 구현·유지보수 난도 중간.
- 단점: 시트가 길어질 수 있음 → 단계 수 자체를 질문 예산으로 제한해 해결.

### C안 — 전부 펼친 폼 (v0.1.0 방식 개선판)
- 장점: 구현 최소.
- 단점: 감사 문제 #1·#2(과밀·긴 스크롤)를 해결 못 함. 320px·아주 크게에서 저장 버튼이 밀려남. 탈락.

### 선택 근거 요약
70대 이해(요약이 항상 보임) + 젊은 사용자 속도(전환 없음, 최근/지난번 칩과 결합) +
로봄 패밀리 문법(시트·카드·칩 재사용) 모두에서 B안이 우위. A안은 맥락 상실, C안은 과밀.

## 시간 선택 결정

12시+6분 버튼 상시 노출(v0.1.0)을 폐기하고:
- 기본: **큰 현재 선택 시간 표시** + 문맥형 빠른 칩 최대 4개(`30분 뒤` `1시간 뒤` `저녁 7시` `시간 없이` 등,
  일정 종류·현재 시각·최근 사용에 따라 변경) + `[시간 바꾸기]`.
- 시간 바꾸기: `[−10분] [＋10분] [정각]` 큰 버튼 + `[시간 직접 선택]` = OS 기본 `<input type="time">`.
- 커스텀 휠·아날로그 시계는 **만들지 않음** — v6 §5-3 합격 조건(스와이프 무관·스크린리더·200%·양대 브라우저)을
  현재 일정 안에서 자동화 검증까지 끝낼 수 없으므로 OS 선택기를 우선한다는 원칙에 따름.

## 시각 정체성 결정

- 라벤더/플럼 팔레트 유지(#7454c8 계열), warm paper 배경 유지 — "의료 앱처럼 차갑지 않게".
- 장식 추가 대신 **위계 정리**: 시트 제목=구체 질문 1개, 완료 요약은 연한 배지행, 현재 질문만 진하게.
- 상태는 색+텍스트 병기(예: 먹었어요 = 체크 아이콘+문구), 이모지·캐릭터 남발 금지.
- 추모(기일) 톤: 축하 문구·경쾌한 색 미사용, 차분한 중립 문구.

## 토큰 정리

styles.css `:root`의 기존 토큰을 유지·확장: `--cal(=--plum)` 계열, radius(24/18/16), 버튼 최소 높이(52/60),
danger(--sun)·warning(--amber) 상태색, focus ring(3px --plum). 화면별 임의 색·크기 직접 입력 금지 원칙.
