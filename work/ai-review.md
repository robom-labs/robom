# 로봄 패밀리 AI 교차검토 누적 기록

## 2026-07-17 Codex 구현과 Claude Code 독립 검토

### 자격증봄

- 문제와 근거: 공식 세부 준비물이 없는 시험은 사실상 빈 상태였고 월별 일정 탐색 전용 화면이 없었다.
- 사용자 영향: 시험 직전에도 무엇을 준비할지 알 수 없고 여러 시험의 날짜 관계를 한눈에 보기 어려웠다.
- 모바일 영향: 준비물과 일정 확인을 위해 상세·찾기 화면을 반복 이동해야 했다.
- 난이도와 위험: 일반 안내를 공식처럼 보이게 만들 위험과 기존 체크 ID migration 위험이 있었다.
- 판단: 일반 안전 준비물 fallback, 출처 신뢰 수준, 시험 달력, 공통 회차 묶음을 채택했다. 날짜 추정과 비공식 정보의 공식 표시는 제외했다.

### 캘린더봄

- 문제와 근거: 날짜 탭이 곧바로 하단 편집기를 열어 일정 확인보다 입력을 강제했고, 키보드와 하단 메뉴가 편집기를 가릴 수 있었다. 병원·약·약속에 임의 시간이 제안됐다.
- 사용자 영향: 기존 일정 확인과 새 일정 입력의 목적이 섞이고 원하지 않는 시각을 저장할 수 있었다.
- 모바일 영향: 작은 화면에서 입력·시간 변경·저장 CTA 접근이 불편했다.
- 난이도와 위험: localStorage schema와 알림 계약을 바꾸면 데이터 회귀가 크므로 UI 상태만 재설계했다.
- 판단: 날짜 선택과 편집 시작을 분리하고 휴대폰 전체 화면·태블릿 패널·시간 미정을 채택했다. 저장 schema 재작성은 제외했다.

### Claude Code 보완 의견

- 발견: 캘린더 두 칸 breakpoint가 760px라 600×960 소형 태블릿 요구가 빠졌다.
- 검토: 600px에서 두 칸으로 바꾸면 각 열이 좁아질 수 있어 200% 확대와 긴 문자열을 함께 재검증해야 했다.
- 판단: 의견을 채택하되 `minmax(0, …)`와 600×960·200% 회귀검사를 함께 추가했다.
- 결과: CalendarBom 0.6.1 `96a10b8`에 반영했다.

### 다른 담당 관점의 반대·보완

- 데이터 관점은 준비물 출처가 불명확할 때 빈 상태보다 일반 안내가 유용하다는 데 동의하되, `공식`, `교차 검증`, `일반 안내`를 섞지 않도록 요구했다.
- 접근성 관점은 날짜 탭과 일정 추가를 분리하면 단계가 하나 늘지만 오작동과 focus 이동을 줄여 전체 과업은 쉬워진다고 판단했다.
- 운영 관점은 네이티브 회전·태블릿 지원을 다섯 앱에 적용하되 한 번에 공통 런타임을 도입하지 않고 앱별 Expo 설정과 검증기로 제한했다.
- 성능 관점은 새 캘린더 라이브러리와 거대한 공통 UI 패키지를 도입하지 않고 기존 DOM·CSS·core 함수를 재사용하도록 요구했다.
